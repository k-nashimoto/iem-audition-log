# CATALOG.md — 試聴リスト メンテナンスガイド

試聴リスト（曲リスト・カテゴリ）は `src/data.js` の **`TRACKS`（source of truth）** と **`CATEGORIES`（カテゴリ定義）** で管理する。
評価データ（`session.ratings` 等）は各曲の**安定ID**を参照するだけなので、曲の追加・削除・並べ替え・サブ観点の付け替えをしても**過去の評価は壊れない**。
（設計背景は `CLAUDE.md §4`、決定ログは `ROADMAP.md`）

---

## 1. データ構造

- **`CATEGORIES`**：カテゴリ定義（メタ）。`{no, title, pri, point, subs?, hint?}`
- **`TRACKS`**：曲リスト＝source of truth。`{id, cat, sub?, core?, t, a}`
- 以下は**自動生成**なので手で触らない：
  - `CATS`（CATEGORIES＋所属TRACKSの結合ビュー）／`CATALOG_IDS`（現行id集合）／`SUB_LABELS`・`subLabel()`（コード→ラベル）／`TOTAL`（総曲数）

---

## 2. 曲（TRACKS）の記載ルール

| フィールド | 必須 | ルール |
|-----------|------|--------|
| `id` | ✔ | **安定slug**（英小文字＋ハイフン、例 `first-love`）。**一意・不変**。評価はこの id をキーに保存するため、**既存 id は絶対に変更・使い回さない**。 |
| `cat` | ✔ | 所属カテゴリ番号 `"00"`〜`"07"` または `"EX"`。 |
| `sub` |  | サブ観点コード。単一（例 `"01a"`）または連結（例 `"01b→05b"`, `"05a+05b"`, `"05a/05b"`）。区切り記号 `→ + /` は表示にそのまま残る。使うコードは所属カテゴリの `subs` に定義されていること。 |
| `core` |  | `true` で **★店頭コア7**（店頭一次スクリーニング用）。目安は7曲程度に絞る。 |
| `t` | ✔ | 表示タイトル。**書式は「曲名 — アーティスト」**。区切りは **半角スペース＋全角ダッシュ `—`（U+2014）＝ ` — `**。 |
| `a` | ✔ | **注目ポイント**（試聴時に何を聴くか）。※アーティスト名ではない。 |

### `t`（タイトル）の注意
- UI は最初の ` — `（全角ダッシュ）で**曲名／アーティストに分割**して表示する。
- **曲名内にハイフンを使う場合は半角ハイフン ` - ` を使う**（区切りの全角ダッシュと混同させない）。
  - 例：`カタオモイ - From THE FIRST TAKE — Aimer` → 曲名「カタオモイ - From THE FIRST TAKE」／アーティスト「Aimer」
- 全角ダッシュが無い場合は全体を曲名として扱い、アーティストは非表示。

### 著作権
- 曲は**曲名・アーティストの参照のみ**。歌詞など著作物の本文は `a` にも埋め込まない。

---

## 3. カテゴリ（CATEGORIES）の記載ルール

| フィールド | ルール |
|-----------|--------|
| `no` | `"00"`〜`"07"` / `"EX"`。**メイン 00–07 の番号と意味は不変**（既存記録と互換）。 |
| `title` | 表示名。 |
| `pri` | `[ラベル, クラス]` または `null`。クラスは `top`（最重視）/`sens`（重要・感度）/`axis`（決定軸）/`seal`（装着）。機能色に対応。 |
| `point` | 確認観点の説明（1行）。 |
| `subs` | `[{code, label, desc}]`。`code` は `"<no><英字>"`（例 `04a`）、`label`＝表示名、`desc`＝凡例。`TRACKS.sub` と `code` を一致させる。 |
| `hint` | 任意の補足（例：04 の「◎は3タイプすべて○以上」）。 |

---

## 4. 評価・スコアの前提

- 評価記号：`◎○△✕`（`RATES`）、スコア化：`◎4 / ○3 / △2 / ✕1`（`SCORE`）。
- **1曲＝1評価**。サブ観点評価＝その曲の評価（タグ）。
- カテゴリ評価＝**所属曲評価の平均**（別枠の手動メインスコアは持たない）。

---

## 5. 改訂手順（チェックリスト）

1. **`TRACKS` を編集**（追加／削除／並べ替え／`sub`付け替え／文言修正）。
2. **曲を追加**：新しい**安定 `id`（slug）**を付与。既存 id とは重複させない。
3. **曲を削除／差し替え**：`TRACKS` から消すだけ。過去のその曲の評価は**孤児評価**として保持され、集計からは除外される（`CATALOG_IDS` で判定、詳細画面に件数注記）。
4. **カテゴリ／サブ観点の増減**：`CATEGORIES` を編集し、`subs.code` と `TRACKS.sub` を一致させる。
5. **`CATALOG_VERSION` を上げる**（例 `"2.2"` → `"2.3"`）。以後の新規採点セッションに版が記録される。
6. **反映確認**：`http` 配信で開いて表示・集計・進捗（`TOTAL`）が正しいか確認 → commit & push（GitHub Pages 反映）。
7. 必要なら **`sw.js` の `CACHE` 版を上げる**とオフライン端末にも確実に更新が届く（`data.js` は ASSETS 登録済みなので、曲編集だけならファイル増減はなし）。

### `OLD_ID_MAP` について
- 旧・位置ベースID（`"01-0"` 等）→ 新・安定ID の**移行表**。`null` は「その評価を破棄」。
- これは **v2.2 移行専用の歴史的マップ**。通常の曲改訂では**触らない**（安定IDを使う限り不要）。

---

## 6. やってはいけないこと

- 既存 `id` の**変更・使い回し**（過去評価が消える／誤って別曲に紐付く）。
- `t` の区切りに**半角ハイフン**を使う（曲名／アーティストの分割が崩れる）。
- メインカテゴリ **00–07 の番号・意味の変更**（既存記録と非互換になる）。
- 別枠の**手動メインスコア**の追加（カテゴリ評価は曲評価の平均、という設計に反する）。
- 外部CDN/フォント/API への依存追加（オフライン維持）。
