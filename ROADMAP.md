## フェーズ1：今の単一HTMLの強化（すぐ着手可・低リスク）

単一ファイル＋localStorage のまま実現できる範囲。

- [ ] **機種横断の比較ビュー**: 複数 session を選び、8カテゴリ×評価(◎○△✕)をマトリクスで一覧。
- [ ] **可視化**: カテゴリ別のレーダーチャート/バーで機種比較（◎=4…✕=1 等のスコア化）。外部ライブラリは極力使わず素のSVG/Canvasで（オフライン維持）。
- [ ] **サマリー**: session ごとに「◎の数・平均スコア・要注意(✕/△)カテゴリ」を自動集計。
- [ ] **エクスポート整備**: JSON に加え CSV 書出（表計算で比較したい用途）。
- [ ] **検索/並べ替え**: 一覧を機種名・日付・スコアでソート/フィルタ。
- [ ] **品質**: localStorage 満杯/破損時のフォールバック、リセットの粒度（全消去/session単位）。

完了条件: 「Kronos vs VE10 vs 他機種」を1画面で比較できる。

## フェーズ2：IEM情報の検索・蓄積（要・設計判断）

「世の中のIEM情報」をどう持つかが論点。自動収集は規約・安定性の問題があるため、**自分で育てるローカルDBから**。

- [ ] **IEMマスタのデータモデル設計**: `{ id, brand, model, price, driverConfig, signature, notes, links[], owned, status }` 等。
- [ ] **手入力＋メモ**でIEM情報を蓄積（このスレッドで作った Noble/VE/Forte Ears 等の情報を初期データに）。
- [ ] session と IEMマスタを紐付け（機種名フリーテキスト → マスタ参照へ）。
- [ ] 参考リンク集（レビュー/中古相場/メーカー）をIEMごとに保持。
- [ ] （任意・半自動）検索補助: 公式ブログ/レビューサイトのURLを貼ると要点をメモ欄に整形、等。フルスクレイピングは規約・安定性から避け、人手レビュー前提で。
- [ ] **アーキテクチャ判断**: ここで単一HTML→「静的サイト（複数ファイル/簡易フレームワーク）＋localStorageまたはIndexedDB」への移行を検討。必要なら軽量バックエンド（データ共有・端末横断同期が欲しくなった場合）。

決定が必要: ローカル完結を続けるか、クラウド同期を入れるか。→ まずローカル完結で。

## フェーズ3：Qobuz / Apple Music 連携（要・API調査・最も重い）

外部API・認証が絡む。実現性調査から。

- [ ] **Apple Music**: MusicKit JS の調査（Apple Developer 登録・デベロッパートークン・ユーザートークンが必要）。プレイリスト読込→チェックリスト自動展開、Web上での再生連携が現実的なゴール。
- [ ] **Qobuz**: 公式公開APIが限定的。実現可能な範囲（メタデータ取得/プレイリスト連携の可否）をまず調査。難しければ「手動同期＋リンク」に留める。
- [ ] **プレイリスト→チェックリスト自動マッピング**: 連携で取得した曲を、好みプロファイルに基づき8カテゴリへ半自動割当（最終は人手調整）。
- [ ] 認証情報・トークンの安全な扱い（クライアントのみで完結しない場合はバックエンド必須＝フェーズ2のアーキ判断と連動）。

注意: ここはオフライン前提と相反する。連携機能は任意・オンライン時のみ動く形にし、コア（試聴ログ）はオフラインで完結させ続ける。
